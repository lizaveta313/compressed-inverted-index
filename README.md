# Сжатый инвертированный индекс

Проектно-практическая работа по теме: «Разработка и экспериментальное исследование сжатого инвертированного индекса с использованием дельта-кодирования и гамма-кодирования Элиаса».

## Что сделано

В проекте реализованы:

- обычный инвертированный индекс;
- сжатый индекс для postings lists;
- дельта-кодирование `doc_id`;
- гамма-кодирование Элиаса;
- сравнение обычного и сжатого индекса по времени и размеру;
- проверка поиска по запросам «Ректор СПбГУ» и «Ректор МГУ».

Для экспериментов используются документы из второго модуля и сгенерированные данные для запуска на большем количестве документов.

## Запуск эксперимента

Базовый запуск на сгенерированных документах:

```bash
python -m compressed_inverted_index.experiment
```

Benchmark на данных из второго модуля:

```bash
python -m compressed_inverted_index.benchmark --data data/module2/spbu_documents.jsonl --sizes 10 25 50
```

Benchmark на сгенерированных документах:

```bash
python -m compressed_inverted_index.benchmark
```

После запуска результаты сохраняются в:

- `reports/benchmark_results.csv`;
- `reports/experiment_report.md`.

## Формат данных

Файл из второго модуля лежит здесь:

```text
data/module2/spbu_documents.jsonl
```

Это JSONL-файл: одна строка соответствует одному документу. При чтении берётся текст из поля `text`, `content` или `body`.

Пример:

```json
{"url": "https://spbu.ru/...", "text": "текст страницы"}
{"id": 123, "content": "текст публикации"}
```

## Кратко про сжатие

В обычном индексе для каждого слова хранится список документов, где оно встречается. Например:

```text
ректор -> [0, 3, 7, 10]
```

В сжатом индексе сначала сохраняются не сами `doc_id`, а разности между ними:

```text
[0, 3, 7, 10] -> [1, 3, 4, 3]
```

Первый `doc_id` записывается как `doc_id + 1`, потому что гамма-код Элиаса работает только с положительными числами. После этого полученные числа кодируются гамма-кодом Элиаса.

## Результаты

Основные результаты приведены в отчёте:

```text
reports/experiment_report.md
```

На данных второго модуля проверялись корпуса на 10, 25 и 50 документов. Для 50 документов сжатый индекс занял меньше места, а результаты поиска совпали с обычным индексом.

Отдельно запускался benchmark на сгенерированных данных до 40 000 документов. В нём также проверялось совпадение результатов для обычного и сжатого индекса.
